Projet : jeu de cartes tactique sur plateau hexagonal

Moteur : Unity

Date : 09 juin 2026

Etat : premiere moitie du rapport complet

1. Introduction

Ce rapport presente l'etat actuel du projet de jeu, ses fonctionnalites principales, son architecture technique et les premiers diagrammes UML necessaires a la documentation finale. Le projet est un jeu tactique tour par tour combinant un systeme de cartes, une grille hexagonale, des unites, des effets de terrain, des sorts, une interface utilisateur et un adversaire controle par l'ordinateur.

L'objectif de cette premiere partie est de documenter les fondations du jeu : boucle de partie, gestion des joueurs, systeme d'achat, main de cartes, plateau, pipeline de jeu des cartes et premiers elements d'intelligence artificielle.

2. Presentation generale du jeu

Le jeu oppose deux camps : le joueur humain et un ennemi controle par l'IA. Chaque joueur possede un fort, une main de cartes, de l'argent et des cartes pouvant etre achetees, jouees, deplacees ou utilisees selon leur type. Le but principal est de reduire les points de vie du fort adverse a zero.

  • grille hexagonale representant le champ de bataille ;
  • cartes de personnages devenant des unites sur le plateau ;
  • cartes de sorts appliquant des effets immediats ou temporaires ;
  • cartes d'effets de monde modifiant des cases du plateau ;
  • gestion de ressources permettant d'acheter de nouvelles cartes ;
  • boucle de tour : phase d'achat puis phase de jeu ;
  • IA capable d'acheter, de jouer, de deplacer et d'attaquer.

3. Fonctionnalites deja presentes

3.1 Gestion de partie

Le script central GameManager gere la creation de la partie, l'etat courant, les joueurs, les phases, les achats, les defausses, les tours et la detection de fin de partie. Il reference GameConfig, CardLibrary, HUDManager, ComputerPlayer et HandUI.

3.2 Phases de jeu

Le jeu utilise une enumeration GamePhase. Les phases principales sont Buy, Play et GameOver. Le comportement des boutons depend de la phase pour guider le joueur et eviter les actions hors contexte.

3.3 Achat de cartes

Le systeme d'achat permet de choisir un cout entre 2 et 7. Le bouton Buy ouvre un menu affichant $2, $3, $4, $5, $6 et $7. Une carte aleatoire du cout choisi est ensuite creee depuis la bibliotheque.

Si la main est pleine, le jeu entre dans un etat de decision : confirmation de l'achat puis defausse obligatoire.

3.4 Main, defausse et zones de cartes

Les cartes sont representees en runtime par CardRuntimeState. Une carte peut etre dans les zones Deck, Hand, Board ou Discard.

3.5 Plateau hexagonal

Le plateau est genere par HexGrid a partir d'un prefab de case hextile. Chaque case possede un composant HexTile, avec son type, son proprietaire, son occupant et son effet de monde eventuel.

3.6 Unites et combats

Les cartes de personnage deviennent des unites sur le plateau. Le UnitManager gere la selection, les deplacements, les attaques, les surlignages et certaines animations de mouvement.

Le projet suit une architecture orientee composants Unity. Les MonoBehaviour gerent les objets en scene, tandis que les ScriptableObject stockent les donnees configurables des cartes et de la partie.

6. Pipeline de cartes

  1. Le joueur selectionne une carte et une cible.
  2. CardPlayService verifie que la partie est en phase Play.
  3. Le service resout le joueur actif et son adversaire.
  4. Le validateur verifie si la cible est autorisee.
  5. Le pipeline applique l'effet de carte.
  6. Le writer modifie l'etat reel : main, plateau, unite, fort, effet de monde.
  7. La carte finit soit sur le plateau, soit dans la defausse.

Cette architecture limite les dependances directes entre UI et logique de carte. Elle facilite aussi l'utilisation du meme systeme par l'IA et par le joueur humain.

7. Intelligence artificielle

L'IA est representee par ComputerPlayer. Elle synchronise son etat avec le GameManager, puis execute une boucle de tour avec un delai entre les actions. La logique repose sur ComputerBrain, ComputerGameSnapshotProvider, LegalActionGenerator, ComputerActionExecutor et LegalActionService.

  • jouer une carte personnage ;
  • jouer un effet de monde ;
  • jouer certains sorts ;
  • deplacer ses unites ;
  • attaquer une unite ennemie ;
  • attaquer le fort adverse ;
  • terminer son tour.

8. Interface utilisateur

L'interface est composee du panneau joueur, du panneau ennemi, du panneau de round, du panneau d'annonce, du plateau central, de la main et des boutons contextuels. HUDManager met a jour les valeurs visibles : points de vie des forts, argent, nombre de cartes, joueur actif, messages d'erreur, annonces de sorts et ecran de fin de partie.

  • en phase achat : Buy, Discard, Play ;
  • si la main est pleine et qu'un achat est confirme : uniquement Discard ;
  • en phase jeu : End Turn ;
  • pendant le tour IA : controles joueur caches ou desactives.
